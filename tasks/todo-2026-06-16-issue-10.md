# Issue #10 — JwsParser drops the verified signer kid when it is only in the unprotected header

GitHub: https://github.com/moisesja/dataproofs-dotnet/issues/10
Release: bump to v1.0.1

## Problem

`JwsParser` resolves the verifying key from the per-signature **unprotected** header
`kid` and verifies the signature against it, but then returns only the **protected**
header `kid` as `JwsParseResult.SignerKid`. When a producer (DIDComm v2.1) places the
`kid` only in the unprotected header, `SignerKid == ""` despite a successful
verification — breaking the downstream addressing-consistency / `from` binding.

This reverses a prior over-conservative decision (issue #6, Finding #2), which the
issue #10 analysis shows was unnecessary: a forged unprotected `kid` resolves to a
different key under which the signature cannot verify, so the `kid` that produced a
*successful* verification is authentic and correct to report. The protected-vs-
unprotected agreement check (reject when both present and differ) is retained.

## Acceptance criteria (from the issue)

- Flattened/general JWS with `kid` only in the unprotected header, that verifies →
  `SignerKid` equals that kid (not `""`).
- `kid` in the protected header → unchanged.
- Conflicting protected vs unprotected `kid` → still rejected.

## Review

**Outcome:** Fixed. `JwsParser` now returns the kid that resolved the verifying key
(`verifiedKid = protected ?? unprotected`) after a successful verification.

**Changes:**
- `src/DataProofsDotnet.Jose/Signing/JwsParser.cs` — verified-kid sourcing + rewritten
  security comment + `SignerKid` XML doc.
- `tests/.../Hardening/HardeningRegressionTests.cs` — reversed the issue-#6 Finding #2
  test (`...VerifiesAndReportsTheResolvedSignerKid`), added a conflicting-kid-rejected
  test and a General-serialization unprotected-only test.
- `Directory.Build.props` — 1.0.0 → 1.0.1.
- `CHANGELOG.md` — `## [1.0.1]` entry.

**Verification:**
- Full Jose suite: **313/313 pass** (incl. the 3 reversed/added regression tests).
- Adversarial subagent review: **NOT EXPLOITABLE** — 7 concrete attacks (kid aliasing,
  multi-sig forged kid, empty/duplicate/case/whitespace kid, compact path) all stopped by
  the agreement check (reachable in both flattened + general paths), post-verification
  ordering, strict JSON, and the fail-closed catch blocks.

**Deliberately out of scope (noted for follow-up):** the reviewer's optional defense-in-depth
suggestion — assert the resolved `publicJwk.Kid` (when present) equals the requested kid to
neutralize a *kid-ignoring* resolver. Pre-existing, not exploitable for the DIDComm consumer
(whose resolver is injective), and risks breaking legitimate fixed-resolver callers; left out
to keep this fix minimal.
