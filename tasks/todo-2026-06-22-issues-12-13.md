# Issues #12 + #13 → v1.1.0 (bundle)

Branch: `fix/issues-12-13-jwe-constant-work-ecdhkey`. Plan approved via plan mode
(`.claude/plans/delegated-dazzling-platypus.md`).

## Scope

- **#12 (`bug`)** — timing side-channel: `JweParser.Parse` fast-failed before any ECDH when no
  recipient `kid` matched a held key → recipient-key enumeration oracle. **Valid** (confirmed in
  source). Note: the issue's own suggested fix was incomplete — it missed the held-but-wrong-curve
  leak and (as the adversarial review later showed) the exception-channel leak.
- **#13 (feature, not a bug)** — additive async `IEcdhKey` seam for opaque (HSM/keystore) ECDH keys.
  **Valid & sound**; NetCrypto 1.1.0 raw-`Z` `IKeyStore.DeriveSharedSecretAsync` confirmed present.

## What shipped

### #12 — constant-work + uniform-failure (internal only, no public API change)
- [x] `DecoyKeyCache` — per-process **cached** decoy private key per curve (+ oct KEK). Caching is
      load-bearing (per-call keygen would re-introduce a timing signal).
- [x] `JweParser.SelectRecipientOrDecoy` — decision keys on *"held key matching the envelope `epk`
      work curve, with private material"*, else routes through the decoy and runs the same
      ECDH→unwrap, failing uniformly. Closes the possession **and** held-wrong-curve leaks.
- [x] **Uniform failure** (`DecryptFailureMessage`) — all post-selection unwrap/AEAD failures throw
      one `JoseCryptoException`, same message, no kid, no inner cause (sync + async). Closes the
      exception-type/message channel the adversarial review found.
- [x] Always-on (secure default), no opt-out. `ParseCompact` (single-key) untouched.

### #13 — additive async `IEcdhKey` seam (sync API + vectors unchanged)
- [x] Public `IEcdhKey` (`Crv` + `DeriveAsync`→raw `Z`) and `RawEcdhKey`.
- [x] `JweParser.ParseAsync`/`ParseCompactAsync`, `JweBuilder.BuildEcdh1PuA256KwAsync`.
- [x] Async KDF cores in `EcdhEsKdf`/`Ecdh1PuKdf` sharing a new `FinishConcatKdf` with the sync
      cores (frozen draft-04 / RFC 7518 vectors untouched). Shared `ResolveSender` extraction.
- [x] PublicAPI.Unshipped.txt additions; `DataProofsVersion` 1.0.1 → 1.1.0; CHANGELOG `[1.1.0]`.

## Review / verification

- **Tests:** 818/818 across the solution (Jose 337, incl. 6 #12 + 16 #13 + uniform-failure +
  public-only-key regressions). Release build clean (0 warnings, `TreatWarningsAsErrors`), so the
  RoslynApiAnalyzer + BannedSymbols analyzers pass. Conformance vectors stay green (KDF extraction
  is byte-identical).
- **Adversarial review (2 passes):**
  - Pass 1 found **Finding 1 (Medium)** — timing equalized but exception type/message still leaked
    possession. Fixed via uniform failure + a regression test.
  - Pass 2 verified Finding 1 fully closed (12/12 probes: held == unheld exception across iv/tag
    length, garbled ciphertext, 1PU, async) and no regression (zeroization intact, broad catch can't
    swallow null args since the wrapped calls take `ReadOnlySpan<byte>`, structural pre-selection
    errors keep descriptive messages). One Informational residual (public-only held key) hardened in
    `IsUsableForEnvelope` (`&& !string.IsNullOrEmpty(key.D)`).
- **Documented scope caveat:** the parser's *own* post-resolution path is constant-work for a fixed
  `(alg, enc, epk-curve)`; a fully constant-time decrypt also needs the caller's
  `IJweRecipientKeyResolver` to be timing-independent (stated in XML docs + CHANGELOG).

## Lesson captured
`tasks/lessons.md` — a side-channel fix must close **every** observable channel (time, exception
type/message, inner cause), and the issue's "recommended fix" clauses are a checklist.
