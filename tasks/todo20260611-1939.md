### Adversarial security review (4 parallel agents, live PoCs)
**Verified sound under attack:** signature/tamper detection and re-canonicalization, alg-confusion
and `alg=none` rejection, DER-smuggling resistance (P1363-strict ECDSA verify), AEAD integrity,
ECDH-1PU sender authentication and `apv` recipient-commitment, invalid-curve/point-at-infinity
rejection, controller authorization (incl. the `INVALID_VERIFICATION_METHOD` vs proofPurpose-field
distinction), proof sets/chains, JWK thumbprints, AC-8 no-key-surrender, the AC-6 BCL-crypto ban,
constant-time `sd_hash`/MAC comparison, CSPRNG randomness and determinism, and exception-message
hygiene (no key/plaintext leakage).

**Findings fixed (130d5f4) — all "hostile input must fail closed, not throw" (FR-3/FR-23), + 1 COSE conformance:**
1. (HIGH) Core pipeline crashed on a duplicate top-level JSON member → now `PROOF_TRANSFORMATION_ERROR`.
2. (HIGH) JOSE JWS/JWT verify leaked `CryptographicException`/`ArgumentException` on an off-curve/malformed verifier JWK → now fails closed.
3. (HIGH) SD-JWT KB-JWT out-of-range `iat` was an unauthenticated DoS → now a structured failure.
4. (HIGH) COSE verify threw on a malformed/wrong-length public key → now `InvalidSignature`.
5. (MED) JWT out-of-range NumericDate threw `ArgumentOutOfRangeException` → now `MalformedJoseException`.
6. (LOW/MED) COSE accepted a `crit` label whose parameter was absent → now rejected (RFC 9052 §3.1).
18 regression tests added, each proven to throw/mis-verify against the unpatched path.

**Investigated and declined (false positive):** COSE "`alg` honored from the unprotected bucket"
— RFC 9052 §3.1 permits an unprotected-only `alg`, the authoritative cose-wg `sign-pass-01`
positive vector relies on it, and cross-bucket duplicates are already rejected, so there is no
forgery. Applying the proposed rejection would have broken interop and an existing passing
conformance test.

**Findings escalated then RESOLVED:** bbs-2023 mandatory-disclosure was not cryptographically
enforced because NetCrypto v1's `IBbsCryptoProvider` did not expose the BBS `header` parameter, so
a malicious holder could drop a mandatory claim and still verify. Filed **moisesja/crypto-dotnet#2**
(root cause: the native FFI already accepted the header; the managed API hardcoded it empty).
**Resolved (2026-06-13, commit 42f542e):** NetCrypto 1.0.0-preview.2 shipped the `header`
parameter; the suite now binds `bbsHeader = SHA-256(proofConfig) ‖ SHA-256(mandatory N-Quads)` at
sign and derive and **recomputes it at verify from the revealed mandatory messages**, so any
drop/alter/reclassify of a mandatory statement fails verification. Regression test
`Verify_MandatoryStatementReclassifiedAsSelective_FailsClosed`; tracker marked RESOLVED; the
interim limitation notes were removed.

### Known deviations from the PRD
1. **bbs-2023 mandatory binding — RESOLVED** (NetCrypto 1.0.0-preview.2, commit 42f542e). bbs-2023
   proofValues remain not byte-interchangeable with W3C `vc-di-bbs` reference vectors because the
   blank-node relabeling keys the HMAC on document-order-stable skolem ids rather than canonical
   `c14n*` labels (validated structurally; a separate, non-security interop item).
2. **Randomness source** — `JoseCryptoProvider.Fill` uses the platform CSPRNG
   (`RandomNumberGenerator`) for SD-JWT salts/IVs because NetCrypto v1 exposes no public randomness
   API; outside NFR-5's "NetCrypto's randomness" letter but the correct secure choice and not a
   banned primitive. Reroute if NetCrypto adds a SecureRandom API.
