## 3. JwsParser behavior contract (subtle, must survive the port)

Order of operations per signature entry (first verifying signature wins; failures accumulate into `lastFailure`, the last one is rethrown if none verifies; if zero resolvable kids → `CryptoException("No JWS signature verified.")`):

1. Resolve public JWK by unprotected-header `kid`; `null` → skip with `CryptoException($"No verifier public JWK supplied for signer kid '{sig.Kid}'.")` recorded.
2. Decode protected header.
3. **`crit` rejection (RFC 7515 §4.1.11):** any `crit` member in the extension bag → `MalformedMessageException("JWS protected header marks an unsupported extension critical ('crit').")` — the implementation understands no extensions.
4. **`b64=false` rejection (RFC 7797):** `AdditionalMembers["b64"].ValueKind == JsonValueKind.False` → `MalformedMessageException` ("JWS with b64=false (unencoded payload, RFC 7797) is not supported."). I.e. **no detached/unencoded payload support today** — FR-13's detached-payload support is new work, not a port.
5. **Alg↔key binding:** `header.Alg` must equal `KeyTypeMapper.ToJwsAlgorithm(publicJwk.Crv)` (ordinal) — prevents algorithm-confusion (e.g. an ES256 signature checked against an Ed25519 key). Mismatch → `CryptoException`.
6. **kid placement rule:** kid may live in protected or unprotected header; spec vectors carry it only unprotected. When both present they MUST match (`MalformedMessageException` on mismatch); when only protected has it, `ExtractSignatures` falls back to `JwsProtectedHeader.Decode(protB64u).Kid`.
7. Verify over `ASCII(protectedB64u + "." + payloadB64u)` with `cryptoProvider.Verify(header.Alg, publicKeyBytes, signingInput, sig.Signature)`; public key bytes via `NetDid.Core.Jwk.JwkConverter.ExtractPublicKey(JwkConversion.ToNetDidJwk(publicJwk))`.
8. Deserialize payload (DIDComm `Message`), run the FR-CONSIST-03 check (drop per §2e).

Flattened detection: top-level string `signature` **and** string `protected`. General: `signatures` array where each entry must have string `protected` + `signature` (else `MalformedMessageException("JWS signature entry is missing a string 'protected' or 'signature'.")`). Signature base64url decode failure → `MalformedMessageException("JWS 'signature' is not valid base64url.")`.

## 4. JweBuilder / JweParser behavior contract

### Build (both pack paths share `EncryptAndAssemble`)

1. All recipients must share one `crv` (ordinal compare) — else `ArgumentException` mentioning FR-ENC-04/FR-ENC-11.
2. Fresh `EphemeralKeyPair.Generate(curve)`; cleared in `finally`. For authcrypt, the sender's `d` is decoded **before** the ephemeral is generated (deliberate ordering so a malformed `d` cannot leak an un-zeroed ephemeral; see code comment at `JweBuilder.cs:113-115`), and zeroed in `finally`.
3. `apv` = `ApvComputer.ComputeBytes(recipient kids)`; header gets base64url form. Authcrypt additionally: `apu = ApuComputer.Compute(skid)` and **the KDF PartyUInfo is `Encoding.UTF8.GetBytes(skid)` — the base64url-DECODED apu** (1PU draft-04 §2.3; comment at `JweBuilder.cs:123-125`). Anoncrypt PartyUInfo is empty.
4. Header: anoncrypt `{typ, alg=ECDH-ES+A256KW, enc, epk, apv}`; authcrypt `{typ, alg=ECDH-1PU+A256KW, enc, epk, apv, apu, skid}`. Encoded with the deterministic writer; `aad = Encoding.ASCII.GetBytes(protectedB64u)`.
5. CEK (`ContentEncryptionKeySizeBytes(enc)`: 64/32/32) and IV (16/12/24) from `cryptoProvider.Fill` (= `RandomNumberGenerator.Fill`). CEK zeroed in `finally`.
6. **Encrypt first**, then per recipient derive KEK and `KeyWrap("A256KW", kek, cek)`; for authcrypt the just-produced AEAD `tag` is passed into `Ecdh1PuKdf.DeriveKey(...)` (tag-in-SuppPubInfo binding, FR-ENC-15); each KEK zeroed in `finally`. `keyDataLen: 32` always (A256KW).
7. Render General JSON: `{protected, recipients:[{header:{kid}, encrypted_key}], iv, ciphertext, tag}` (all base64url). **No compact JWE serialization exists** — FR-14's compact support is new work.

### Parse

Order matters (tests pin it):
1. Structure parse with `DidCommJson.StrictDocument` (duplicate-member rejection). Required string members `protected`, `iv`, `ciphertext`, `tag`; `recipients` array with object `header.kid` + `encrypted_key`; ≥1 recipient. Any miss → `MalformedMessageException` naming the member.
2. Header decode; **`crit` → `MalformedMessageException`** (RFC 7516 §4.1.13).
3. `enc` allowlist (`IsSupportedContentEncryption`) → else `CryptoException($"Unsupported JWE 'enc' '{header.Enc}'.")`; authcrypt (`alg==ECDH-1PU+A256KW`) additionally pinned to `enc==A256CBC-HS512` → else `CryptoException` (DIDComm FR-ENC-09 — **decide consciously whether dataproofs keeps this restriction**; the 1PU draft itself only requires a key-committing AEAD for the *key-wrap* modes, and A256CBC-HS512 is what draft-04 §2.1 mandates for non-direct mode — keeping it preserves parity).
4. **`apv` re-derivation before any ECDH:** recompute from the envelope's recipient kid list and ordinal-compare to `header.Apv`; mismatch → `CryptoException` mentioning `apv` (FR-ENC-13 tamper check).
5. Recipient selection: `secretsLookup.FindPresent(kids)` then first present; missing → `CryptoException("No recipient kid in the JWE matches a held private key.")`.
6. Recipient key `crv` must equal `epk.crv`; epk extraction via `JwkConversion`/net-did validates the point is on-curve (`CryptographicException` → wrapped as `CryptoException("JWE 'epk' is not on the asserted curve.")`).
7. Authcrypt sender identity: use `skid` when present; else derive from `apu` (`utf8(b64udecode(apu))`); when **both** present they must agree (`ApuComputer.Compute(skid)` vs `header.Apu`) → else `CryptoException("Authcrypt 'apu' does not match base64url(skid) (FR-ENC-14).")`; neither → `CryptoException` (missing both skid and apu). Sender curve must match recipient curve. KDF apu bytes = decoded `apu` (fallback `utf8(skid)`).
8. KEK derive (recipient-side KDF variants), `KeyUnwrap` (`CryptographicException` → `CryptoException("AES-KW unwrap failed ...")`), `AeadDecrypt` (`CryptographicException` → `CryptoException("AEAD decryption failed ('{enc}').")`). Recipient private bytes, KEK, CEK each zeroed in `finally` blocks.

## 10. Risks / PRD-assumption breakers

1. **EC public-key wire shape:** net-did's keygen/JWK converter uses **compressed SEC1** EC points (33/49/67 bytes — pinned by `EphemeralKeyPairTests`). If NetCrypto's `DeriveSharedSecret`/keygen expects uncompressed SEC1 or raw X‖Y, the epk construction, `ExtractPublicKey`, and those length assertions all change. Verify NetCrypto's point encoding before porting; this is the most likely silent interop breaker.
2. **ConcatKdf parameter semantics:** the 1PU SuppPubInfo assembly assumes the KDF length-prefixes algorithmId/partyU/partyV internally but passes suppPubInfo through verbatim (NetDid behavior, documented in its XML docs). If NetCrypto's `ConcatKdf` prefixes differently, every derived KEK changes. Pin with the `Ecdh1PuKdfTests` differential test early.
3. **JWK boundary type:** `JwkConversion` is built around `Microsoft.IdentityModel.Tokens.JsonWebKey` via net-did's `JwkConverter` (which also owns the off-curve validation). dataproofs may keep only `JsonWebKey` itself (AC-7's single allowed type) and must re-home (a) EC on-curve validation and (b) the OKP 32-byte check — if NetCrypto/NetCid don't provide them, they are security regressions waiting to happen, and `JwkConversionTests` will catch only (b).
4. **`CryptographicOperations.ZeroMemory`:** used pervasively; AC-6's allowlist names only `FixedTimeEquals`. Either extend the allowlist (it is a memory utility, not a primitive) or find a NetCrypto equivalent — decide before the banned-symbol scan lands.
5. **Test helper coupling to NSec/NetDid:** `TestKeyMaterial`, `Ecdh1PuKdfTests`, `EcdhEsKdfTests`, `EphemeralKeyPairTests` and the round-trip tests all call `NetDid.Core` types (`DefaultKeyGenerator`, `KeyType`, `DefaultCryptoProvider`, `ConcatKdf`) directly. These are renames under AC-5 (NetDid→NetCrypto symbols), but the parity-diff rename map must cover enum/type names (`KeyType.X25519` etc.) or the diff tool will flag them.
6. **Async surface change:** the source builders/parsers are synchronous and take raw private-key bytes. Moving signing to async `ISigner` (NFR-3/AC-8) changes test call shapes (`Build(...)` → `await BuildAsync(...)`). Assertion content can stay identical, but the parity tool's notion of "asserted structure" must tolerate the await rewrite.
7. **Authcrypt enc pinning (`ECDH-1PU` ⇒ `A256CBC-HS512`)** is a DIDComm rule layered on draft-04. Generic FR-14 JWE could legitimately allow 1PU with other encs in *direct* mode, but draft-04 §2.1 restricts key-wrap mode to the CBC-HMAC family — keep the restriction for key-wrap mode to preserve both spec-compliance and parity assertions.
8. **No compact JWS/JWE, no detached payload, no RFC 7638, no JWT validation exist in the source** — FR-13/14/15 are part-port, part-new-build. Schedule estimates assuming "port covers FR-13/14" will be wrong by roughly the compact/detached/thumbprint/claims surface.
9. **`System.Buffers.Text.Base64Url` dependency** requires .NET 9+; fine on .NET 10 (NFR-1) but the parity tests' `Base64Url` helper calls must keep the no-pad behavior including `DecodeFromChars` strictness (`FormatException` on padded/invalid input — several negative tests depend on it).
10. **DIDComm spec fixtures are not in the Core.Tests tree** — the packed Appendix C envelopes live in `DidComm.InteropTests` behind a DIDComm-specific manifest runner; they are not part of the AC-5 copy set and would need their own vendoring + harness if used (see §7 note).
