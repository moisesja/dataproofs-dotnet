# AC-6 banned-symbol scan — NetCrypto-only crypto (hashing included).
#
# Microsoft.CodeAnalysis.BannedApiAnalyzers reads this file (referenced from each of the five
# src csprojs via <AdditionalFiles Include="$(MSBuildThisFileDirectory)BannedSymbols.txt" />).
# It bans every System.Security.Cryptography CRYPTOGRAPHIC PRIMITIVE — direct BCL hashing,
# HMACs, signature algorithms, and AEADs — that a package-dependency check (the
# tasks/dependency-hygiene tool) cannot see, because these live in System.Private.CoreLib /
# System.Security.Cryptography.* assemblies that ship with the runtime, not as NuGet packages.
# Every such operation MUST instead route through NetCrypto's hashing/signing/AEAD API
# (PRD §2.2 / AC-6 / NFR-6).
#
# Format: one DocID per line, `<symbol-id>;<message>`. Banning a TYPE (`T:`) flags both
# instance construction and static-factory use (e.g. SHA256.HashData(...)). Each declaration's
# `Id` must be its DocumentationCommentId.
#
# DELIBERATELY NOT BANNED:
#   * T:System.Security.Cryptography.CryptographicOperations is NOT banned here, so its
#     FixedTimeEquals constant-time comparison utility stays available (NFR-6 — the single
#     allowlisted symbol from this namespace; it is a comparison utility, not a keyed or
#     algorithmic primitive). Banning the whole System.Security.Cryptography namespace (N:)
#     would have caught it, which is why we ban the primitive TYPES individually instead.
#   * System.Formats.Cbor.* is NOT in System.Security.Cryptography and is the sanctioned
#     internal CBOR encoding for Cose/Rdfc (PRD §2.2) — nothing here touches it.
#   * T:System.Security.Cryptography.RandomNumberGenerator is NOT banned. The AC-6 / PRD §2.2
#     enumerated ban list is hashes + HMACs + signature algorithms + AEADs; randomness is
#     deliberately outside it. NFR-5 bans System.Random (a non-cryptographic PRNG), NOT the
#     platform CSPRNG, and requires "NetCrypto's randomness" — but NetCrypto v1 exposes no
#     public randomness API (tasks/research/netcrypto-api.md §9), so there is no NetCrypto
#     entry point to route to. DataProofsDotnet.Jose.JoseCryptoProvider.Fill therefore uses
#     RandomNumberGenerator.Fill (the platform CSPRNG access point) for SD-JWT salts/IVs; this
#     is a randomness SOURCE, not a keyed/algorithmic primitive, in the same allowlist category
#     as FixedTimeEquals. Recorded as a hygiene NOTE in the Phase F report: if NetCrypto grows
#     a SecureRandom API, reroute JoseCryptoProvider.Fill to it and add the ban line back here.
